# Unify the three dialect enums into one shared `Dialect` (#146) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task (fresh implementer per task + two-stage review; opus for the final whole-branch review). Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Collapse the three parallel `{ Sqlite, Postgres }` dialect enums — `BackendKind`/`SqlDialect` (main crate), `ferro_ddl_lowering::Dialect`, and `ferro_migrate::BackendDialect` — into the **one** canonical `ferro_ddl_lowering::Dialect`, deleting the three hand-written per-seam translators.

**Architecture:** Promote the existing `ferro_ddl_lowering::Dialect` (the leaf crate every consumer already depends on) to the single dialect type; rename all ~270 references to `Dialect`; delete `sql_dialect_to_lowering`, `backend_dialect`, and `lowering_dialect`; relocate `BackendKind`'s one inherent method `from_url` to a free function `dialect_from_url` in the main crate. A short-lived transitional alias keeps the workspace compiling between the per-crate tasks and is removed before the PR lands.

**Tech Stack:** Rust (PyO3, `sea-query` 0.32, `sqlx`), Python (Pydantic), `cargo test` + `pytest` (SQLite + Postgres backend matrix).

**Closes:** [#146](https://github.com/syn54x/ferro-orm/issues/146)
**Base branch:** `feat/ir-p8.6-cleanups` (Epic 8.6 integration branch)

## Key Technical Decisions

### KTD-1: One type, not two (runtime identity is behavior, not a second type)

`BackendKind` also selects the connection driver/pool, but "which dialect I render" and "which database I'm connected to" are the **same** two-variant axis, 1:1 by construction and **bounded by the libraries**: `sea-query` renders exactly `{Sqlite, Postgres, Mysql}` and `sqlx` drives the same three, lined up one-to-one. No driver-≠-dialect divergence is reachable in this stack (a Postgres-wire engine connects as `postgres://` and renders `PostgresQueryBuilder`; `sqlx` has one async driver per backend). So a separate runtime type bridged by an identity map carries zero information. Runtime behavior is preserved as behavior on the one type (KTD-3), not as a second type.

### KTD-2: Canonical home is `ferro-ddl-lowering`

`ferro_ddl_lowering::Dialect` already exists and every consumer (ddl-lowering, ferro-migrate, main) already depends on `ferro-ddl-lowering`. `ferro-schema-ir` (the deeper leaf) is dialect-agnostic and must stay so — do **not** move the type there. The type is **defined once** in `ferro-ddl-lowering`; `ferro-migrate` and the main crate get it by `pub use` / `pub(crate) use`.

### KTD-3: `from_url` becomes a free function, not an extension trait

`from_url` is `BackendKind`'s only inherent method, and Rust forbids inherent `impl`s on a foreign-crate type — so it cannot become `impl ferro_ddl_lowering::Dialect`. It moves to a free function `dialect_from_url(url: &str) -> Result<Dialect, UnsupportedDatabaseUrl>` in `src/backend.rs`, where parsing a connection-URL scheme belongs. A free function (not a one-method extension trait) is correct because there is exactly one such constructor — a trait would be ceremony. `UnsupportedDatabaseUrl` and its exact message stay in the main crate.

### KTD-4: A transitional `BackendDialect` alias keeps the workspace green between tasks

The type change ripples upstream (a crate cannot drop `BackendDialect` while a crate above still names it). To keep the **whole workspace compiling and green at every task boundary** (required by the subagent-driven review gates), Task 3 leaves a one-line transitional `pub type BackendDialect = Dialect;` in `ferro-migrate` so the main crate keeps compiling until Task 4 flips it. Task 5 deletes that alias. It is scaffolding inside this PR, not a stop-gap end state — the final tree has no such alias (Global Constraints).

### KTD-5: Rename mechanics + the three gotchas

The rename is a whole-token identifier replacement (`BackendKind`, `SqlDialect`, `BackendDialect` → `Dialect`) plus deletion of the translator functions, then `cargo build` to surface stragglers, then `cargo test`. Three hazards an implementer/reviewer must watch:

1. **`from_url` call sites must become `dialect_from_url(...)`, NOT `Dialect::from_url(...)`** — the latter won't exist after the relocation. Fix these explicitly in the main-crate task (Task 4) *before* relying on the blanket rename.
2. **Do not touch lookalikes:** `BackendPool` (the pool-wrapper enum) and the `backend` field/`backend()` accessor names **stay** — only the *type* names change. A field `pub backend: Dialect` and `fn backend(&self) -> Dialect` are correct and intended.
3. **The `sea-query` mapping arms are logic, not names** — `match d { Sqlite => SqliteQueryBuilder, Postgres => PostgresQueryBuilder }` keeps its body; only the matched value's type becomes `Dialect`.

## High-Level Technical Design

```
        BEFORE                                          AFTER
 main: BackendKind / SqlDialect                  main: use ferro_ddl_lowering::Dialect
   ├ sql_dialect_to_lowering()  ┐                       (BackendKind/SqlDialect deleted;
   ├ backend_dialect()          ├ 3 translators          from_url → dialect_from_url free fn)
 migrate: BackendDialect         │  deleted        migrate: pub use …::Dialect
   └ lowering_dialect()         ┘                        (BackendDialect deleted)
 ddl-lowering: Dialect                           ddl-lowering: Dialect  ← single source of truth
```

Task order is **bottom-up** so each task leaves the workspace green:
ddl-lowering (Task 2) → ferro-migrate (Task 3, leaves transitional alias) → main crate (Task 4) → remove alias + grep-gate (Task 5) → traceability (Task 6).

/// Infer the SQL dialect / backend from a connection-URL scheme.
