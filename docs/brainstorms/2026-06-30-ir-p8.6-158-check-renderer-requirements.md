# #158 — Single check-renderer for `db_check`

## Goal

Make the `db_check=True` CHECK constraint SQL be **produced in exactly one place
per language**, by carrying the check as **structured data in the SchemaIR**
(`column` + pre-rendered value tokens) instead of a flat, pre-rendered SQL string.
Today the IR carries `SchemaCheck.expression: String` (`role IN ('admin', 'user')`,
unquoted), and three consumers each re-interpret it differently — the create
emitter even re-parses the string positionally to re-quote the column. This is the
epic's "render the same DDL two ways" anti-pattern one level down from #146: the
expression is built in the Python compiler and re-quoted/re-derived in the Rust
emitter.

After this lands, `SchemaCheck` is structured (`{ name, column, values }`), a single
Rust renderer (`render_db_check` in `ferro-ddl-lowering`) serves both the runtime
CREATE path and the `ferro-migrate` ALTER path, the Alembic emitter builds its
`CheckConstraint` body from the same structured fields, and
`render_check_expression`'s positional `split_once(" IN (")` re-quoting is **deleted**.

This is sub-issue #158 of Epic #145 (Phase 8.6), **surfaced by the #153 final
review** (PR #157): #153 added `render_check_expression` as a guard to reconcile the
create path's quoted output with the compiler's unquoted IR string, and explicitly
flagged that the guard is "a smell, not a fix" — a structured shared renderer
deferred to this issue.

## The central design question (resolved: structured IR, one renderer per language)

The issue offers two shapes: (a) the SchemaIR carries structured check data and a
single renderer turns it into SQL, or (b) a shared check-renderer in
`ferro-ddl-lowering` is consumed by every emitter. The wrinkle that forces the exact
shape: **the producer (the Python compiler) and one consumer (Alembic
autogenerate) are both Python, while two consumers (runtime CREATE, `ferro-migrate`
ALTER) are Rust.** A literally-single renderer cannot serve both languages — Alembic
cannot call a Rust function per check, and an FFI hop for one tiny string is ceremony
(and Alembic's empty-diff does not even depend on the column quoting).

**Decision: option (a) — structured IR + one renderer per language.** `SchemaCheck`
becomes `{ name, column, values }`. One Rust renderer (`render_db_check`) serves
CREATE + ALTER; Alembic builds its `CheckConstraint` body from the same structured
fields via a 1-line Python mirror. This kills both halves of the smell: the
positional re-parse **and** the flat-string-in-the-IR. The two body renderers are the
one unavoidable cross-language duplication, but they are reduced to the trivial,
escaping-free, can't-diverge part (`quote(column) + " IN (" + join(values) + ")"`);
the hard part (value-literal escaping) stays single-sourced in the compiler.

### Why `values` are pre-rendered literal tokens, not raw values

`values` carries **already-escaped SQL literal tokens** (`["'admin'", "'user'"]`),
not raw domain values (`["admin", "user"]`). This is the decisive sub-choice. The
value-literal rendering has real, divergence-prone logic that lives in the compiler
today:

```python
if isinstance(value, bool):           rendered.append(str(value).lower())  # True  -> true   (bare)
elif isinstance(value, (int, float)): rendered.append(str(value))          # 5     -> 5      (bare)
else: escaped = str(value).replace("'", "''"); rendered.append(f"'{escaped}'")  # O'Brien -> 'O''Brien'
```

If `values` were raw, **both** the Rust renderer and the Python renderer would have
to re-implement this escaping and stay byte-identical forever — the exact
cross-language "render two ways" this epic kills, merely moved down to the literal
level. Pre-rendered tokens keep escaping in the one place that has the typed Python
enum values; each renderer only quotes the column identifier and joins the tokens.

## Non-goals (explicitly out of scope)

Each has a real path for the excluded case (AGENTS.md I-6 — scoped-down, not
hollowed-out):

- **New check kinds.** Only the `<col> IN (...)` enum-check shape exists today (the
  compiler emits nothing else — pinned by
  `test_schema_ir_compiler_emits_db_check_expression_for_closed_domain`). The
  structured `{ column, values }` shape **deliberately can represent only this
  shape**. A free-form CHECK (e.g. `length(number) > 0`) would need a new IR kind;
  that is a future phase, not this one (YAGNI). See §5 for the one synthetic vector
  this narrowing rewrites.
- **An `expression` escape hatch.** Keeping `expression: Option<String>` "just in
  case" reintroduces the flat-string carrying this issue removes. Rejected.
- **The deprecated `_build_sa_table`** (`src/ferro/migrations/alembic.py`, its own
  ad-hoc `f"{col} IN (...)"`). It is already tests-only with removal slated for
  v0.14.0; the live Alembic autogenerate path is `_build_sa_table_from_ir`. Untouched.
- **Removing the legacy migrate machinery** (`schema_json_to_schema_ir`,
  `migrate_check_expression`, `plan_table_migration_legacy`) — retained for shadow
  comparison until Phase 9 / #108. We update the cfg(test) helpers mechanically to
  the new struct shape; we do not remove them.
- **The dialect-enum unification** — #146, already landed on the integration branch.

## Current state (confirmed on `feat/ir-p8.6-cleanups`)

The IR type carries a flat, **unquoted** string:

```rust
// crates/ferro-schema-ir/src/lib.rs
pub struct SchemaCheck {
    pub name: String,        // "ck_account_role"
    pub expression: String,  // "role IN ('admin', 'user')"   <- unquoted column, rendered values
}
```

The producer and the three live consumers each treat that string differently:

| Site | File / symbol | Role | Emitted CHECK body | Column |
|------|---------------|------|--------------------|--------|
| Producer | `compiler.py::_checks_from_columns` | the one production producer | — (`role IN (...)` into IR) | unquoted |
| Consumer | `emit.rs::post_create_artifacts` (runtime CREATE) | re-quotes via `render_check_expression` (`split_once(" IN (")` + `quote_ident`) | `"role" IN ('admin', 'user')` | **quoted** |
| Consumer | `emit.rs::emit_add_column` (`ferro-migrate` ALTER) | uses `check.expression` raw | `role IN ('admin', 'user')` | **unquoted** |
| Consumer | `alembic.py::_build_sa_table_from_ir` | `sa.CheckConstraint(check.expression)` | `role IN ('admin', 'user')` | **unquoted** |

So the premise is real and slightly worse than the issue states: **CREATE and ALTER
already disagree today** — CREATE quotes the column, ALTER and Alembic do not.
`render_check_expression` re-parses a string the compiler built structured, and only
understands the one `<col> IN (...)` shape (fails loud otherwise — the #153 guard).

**A second, smaller divergence (discovered during this brainstorm):** the SQLite
elision is **loud on CREATE** (`post_create_artifacts` pushes a warning) but
**silent on ALTER** (`emit_add_column` has `if dialect == Postgres { … }` with no
`else`). So adding a `db_check` column to an existing table on SQLite drops the
constraint with no warning — an AGENTS.md I-6 "degrade silently" smell.

There is also a **test-only second producer**: `schema_json_to_schema_ir` (inside
`#[cfg(test)] mod tests` in `src/migrate.rs`) builds `SchemaCheck` via
`migrate_check_expression`, the Phase-9-deferred legacy-vs-IR parity fixture.
Notably it *already* emits the **quoted** form (`format!("\"{}\" IN ({})", …)`) — so
the structured rewrite makes it converge with the compiler by construction.

What is pinned by tests today:

- **CREATE golden** (`crates/ferro-migrate/src/tests.rs`, the `render_create_table`
  golden) asserts the **quoted** Postgres form:
  `ALTER TABLE "account" ADD CONSTRAINT "ck_account_role" CHECK ("role" IN ('admin', 'user'))`.
- **ALTER path** (`emit_add_column`'s check loop) is **untested** — no fixture drives
  a check through it; nothing pins its current unquoted output.
- **Alembic** `db_check` output is **not asserted** in `test_cross_emitter_parity.py`
  (it is an empty-diff test and does not compare CHECK expressions).

## The change

### 4.1 Structured `SchemaCheck` (`ferro-schema-ir`)

```rust
// crates/ferro-schema-ir/src/lib.rs — replaces { name, expression }
pub struct SchemaCheck {
    /// Canonical check name (`ck_<table>_<col>` for single-column checks).
    pub name: String,
    /// The column the check constrains (structured — no longer embedded in a string).
    pub column: String,
    /// Pre-rendered SQL literal tokens for the allowed values, e.g. `["'admin'", "'user'"]`.
    /// Escaping/bool-casing is done once, in the Python compiler; renderers only join.
    pub values: Vec<String>,
}
```

The IR is ephemeral (recompiled each run from the registry, passed over FFI
in-process; never persisted), so there is no stored-IR backward-compat concern. The
`_IR_VERSION` envelope constant (`compiler.py:23`) is an internal contract; the plan
verifies whether the structural change warrants a bump (the contract validator
`_validate_schema_payload` does **not** inspect a check's inner fields, and no test
deserializes the changed shape across a version boundary — so a bump is likely
unnecessary, but the plan confirms rather than assumes).

### 4.2 Producer: compiler emits structured pieces (`compiler.py`)

`_checks_from_columns` keeps its exact value-rendering loop (the escaping above) but
emits the pieces instead of folding them into an f-string:

```python
checks.append({
    "name": f"ck_{table_name}_{col_name}",
    "column": col_name,
    "values": rendered,          # ["'admin'", "'user'"]  (was: "expression": f"{col_name} IN (...)")
})
```

The test-only `migrate_check_expression` / `schema_json_to_schema_ir` (cfg(test),
`src/migrate.rs`) is updated mechanically to set `column` + `values` on the struct.

### 4.3 One Rust renderer (`ferro-ddl-lowering`)

Home: `ferro-ddl-lowering`, next to its siblings `quote_ident` (`:466`) and
`db_check_constraint_name` (`:365`), both already `pub` and dialect-agnostic.

```rust
/// The CHECK body, byte-identical across CREATE and ALTER and (mirrored) Alembic.
/// Quoting is double-quote on both backends; the check only ever emits on Postgres.
pub fn render_check_body(check: &SchemaCheck) -> String {
    format!("{} IN ({})", quote_ident(&check.column), check.values.join(", "))
}   // -> "role" IN ('admin', 'user')

pub struct CheckEmission {
    pub statement: Option<String>,  // the ALTER ... ADD CONSTRAINT ... CHECK (...) (Postgres only)
    pub warning: Option<String>,    // the SQLite elision warning
}

/// Single source for the db_check emission decision: wrapper + dialect + body.
pub fn render_db_check(table: &str, check: &SchemaCheck, dialect: Dialect) -> CheckEmission {
    match dialect {
        Dialect::Postgres => CheckEmission {
            statement: Some(format!(
                "ALTER TABLE \"{table}\" ADD CONSTRAINT \"{}\" CHECK ({})",
                check.name, render_check_body(check))),
            warning: None,
        },
        Dialect::Sqlite => CheckEmission {
            statement: None,
            warning: Some(format!(
                "Check constraint '{}' on table '{}' is not emitted on SQLite (requires table rebuild).",
                check.name, table)),
        },
    }
}
```

### 4.4 Rewire both Rust consumers; delete `render_check_expression`

- **`post_create_artifacts`** (CREATE): replace the inline
  `match dialect { Postgres => format!(... render_check_expression(...)?), Sqlite => warn }`
  with a call to `render_db_check`, pushing `.statement` into `post_create_sqls` and
  `.warning` into `warnings`. (The CREATE byte output is unchanged — already quoted.)
- **`emit_add_column`** (ALTER): replace the `if dialect == Postgres { … }` (no
  `else`) with a call to `render_db_check`, pushing `.statement` and `.warning`. This
  is where the two reconciliations land: the column becomes **quoted** (matching
  CREATE) and SQLite now **warns** instead of silently dropping.
- **Delete `render_check_expression`** (`emit.rs:223`) entirely.

### 4.5 Alembic consumes structured fields (`alembic.py`)

`_build_sa_table_from_ir` reads `check["column"]` + `check["values"]` and builds the
body via a 1-line Python mirror of `render_check_body` (SQLAlchemy builds its own
`CONSTRAINT … CHECK (…)` wrapper, so only the body is shared):

```python
def _render_check_body(column: str, values: list[str]) -> str:
    return f'"{column}" IN ({", ".join(values)})'   # "role" IN ('admin', 'user')
...
sqltext = _render_check_body(check["column"], check["values"])
table_args.append(sa.CheckConstraint(sqltext, name=check["name"]))
```

This changes Alembic's emitted check text from unquoted to quoted. The empty-diff
parity test does not compare CHECK expressions, so it is expected to stay green —
re-verified as a hard gate (§Testing), not assumed.

## Behavior reconciliation (exactly what bytes change)

The canonical form is the **quoted** one everywhere. For `role IN ('admin', 'user')`:

| Path | Today | After | Change |
|------|-------|-------|--------|
| Rust CREATE (Postgres) | `CHECK ("role" IN ('admin', 'user'))` | same | **none** — golden stays green |
| Rust ALTER (Postgres) | `CHECK (role IN ('admin', 'user'))` | `CHECK ("role" IN ('admin', 'user'))` | quoted; now matches CREATE |
| Rust ALTER (SQLite) | *silently dropped* | *dropped + elision warning* | gains the warning |
| Alembic (live IR path) | `role IN (...)` text | `"role" IN (...)` text | quoted; empty-diff re-verified |

CREATE is the only path with a golden, and it is unchanged. The ALTER changes are
reconciliations toward CREATE's existing canonical form (no test pinned the old
bytes). The SQLite-warning addition is the deliberate "fix the divergence you
consolidate" improvement (same posture as #153's inline-SQLite-FK fix), aligned with
AGENTS.md I-6.

## Error handling

`render_db_check` is total — it cannot fail (no parsing, no fallible lookup), which
is the point: the deleted `render_check_expression` returned `Err` precisely because
it re-parsed a string that might not match the expected shape. With structured input
there is no shape to mismatch. No `unwrap()` is introduced across the FFI boundary
(AGENTS.md I-3). The Alembic path keeps its existing defensive `isinstance` guards on
the IR dict fields.

## Risk

- **A behavior change sneaking in under the rewrite.** Mitigation: the CREATE golden
  pins the production path byte-for-byte (unchanged), and the new ALTER test pins the
  reconciled form; the full SQLite+Postgres matrix + cross-emitter parity assert
  end-to-end invariance.
- **Alembic empty-diff flips** from the quoting change. Mitigation: re-verified as a
  hard gate, not assumed; Alembic autogenerate does not deep-compare CHECK
  expressions and Postgres normalizes them, so the column quoting is cosmetic to the
  diff — but the test proves it.
- **A missed consumer of `.expression`.** Mitigation: the grep-gate fails the build
  if any `expression`-on-check reference remains (compiler, emit, Alembic, fixtures,
  cfg(test) legacy helper).
- **The cfg(test) legacy helper** (`schema_json_to_schema_ir`) drifting from the new
  shape. Mitigation: it constructs `SchemaCheck`, so it will not compile until
  updated; it already emits the quoted form, so it converges structurally.

## Migration impact

`none` — internal architecture. Runtime DDL is byte-identical where pinned (the
CREATE production path); the ALTER path converges to the same canonical quoted form
(no prior golden), and the SQLite ALTER path gains a warning where it previously
degraded silently. No user-facing API, type, or surface change (`SchemaCheck` and the
renderers are internal Rust/Python symbols, not part of the public API).

## Documentation impact

`none` — no user-facing API, behavior, or surface change. Justification recorded here
