# #153 вЂ” Unify the CREATE TABLE path onto the Python SchemaIR

## Goal

Make the runtime CREATE TABLE path consume the **Python-compiled SchemaIR**
(`compile_registry_schema_ir`) вЂ” the same declared-schema blueprint the migrate
path already consumes вЂ” instead of re-deriving column types from the raw,
enriched Pydantic JSON. This collapses the **two parallel CREATE TABLE emitters**
(runtime JSON-driven + `ferro-migrate` IR-driven) into **one** IR-driven emitter,
and leaves the Python compiler as the **single declared-schema producer** for both
create and migrate.

After this lands, the deprecated legacy arms in `canonical_from_parts` and the
`#[cfg(test)]`-only `schema_json_to_schema_ir` shim have no remaining caller and
are removed.

**This is a behavior-preserving refactor for runtime DDL. It must emit
byte-identical `CREATE TABLE` + index/constraint SQL on SQLite and Postgres.** The
one intentional behavior change is internal: `ferro-migrate`'s `AddTable` emitter
(not reached in production today) is corrected to emit inline foreign keys on both
backends instead of dropping them on SQLite.

This is the keystone sub-issue of Epic #145 (Phase 8.6) and the last open item
from the 2026-06-26 lowering-consolidation audit: it is what finally makes the
schema/DDL domain single-sourced *by construction* rather than by hand-synced
parity tests.

## Non-goals (explicitly out of scope)

Each has a real path for the excluded case (AGENTS.md I-6 вЂ” scoped-down, not
hollowed-out):

- **Circular foreign keys** (table A в†” B). Inline FKs cannot express a true
  cycle, and SQLite cannot add an FK to an existing table via `ALTER` either.
  This is the *same* boundary that exists today; circular FKs remain Alembic /
  manual-migration territory.
- **The Python Alembic autogenerate emitter** (`src/ferro/migrations/alembic.py`).
  It already derives from the SchemaIR (Phase 4) and is unchanged here.
- **The dialect-enum unification** (`SqlDialect` / `Dialect` / `BackendDialect`)
  вЂ” #146, a separate 8.6 sweep item. We translate at the call seam as today.
- **The duplicated alter-helpers in `src/migrate.rs`** вЂ” #142 / Phase 9.

## Current state (confirmed on `main`)

There are two CREATE TABLE emitters, and they are **not** byte-identical:

**1. Runtime, JSON-driven (production).** Reaches the database through one
chokepoint, `internal_create_tables` (`src/schema.rs:523`), shared by all three
create entrypoints:

- `connect(auto_migrate=True)` в†’ `internal_migrate` (`src/migrate.rs:854`) в†’
  `internal_create_tables` (create always runs first, before any diff).
- `migrate(...)` (pyfunction, `src/migrate.rs:961`) в†’ `internal_migrate` в†’ same.
- `create_tables(...)` (pyfunction, `src/schema.rs:600`) в†’ `internal_create_tables`.

`internal_create_tables` reads `MODEL_REGISTRY` (raw JSON), orders tables via
`order_schemas_for_creation` (`src/schema.rs:94`), and renders each via
`build_create_table_sqls` (`:453`) в†’ `build_column_plan` (`:349`) в†’
`canonical_column_type` (`:221`) в†’ `property_json_type_and_format` (`:30`) в†’
`canonical_from_parts`. That last call feeds the **raw JSON-Schema `type`+`format`
vocabulary** (`"string"` + `"date-time"`, etc.), which is exactly what keeps the
deprecated arms alive. FKs are emitted **inline** inside `CREATE TABLE`,
**anonymously** (no constraint name), on **both** backends (`:473вЂ“481`).

**2. `ferro-migrate`, IR-driven (test/planner-only).** `render_create_table`
(`crates/ferro-migrate/src/emit.rs:53`) + `emit_add_table_passes` (`:237`) build a
`CREATE TABLE` from a `SchemaModel`. **In production this is never reached** вЂ” the
runtime auto-migrate path only ever constructs *column-level* `MigrationPlan`s
(`src/migrate.rs:518,1106`); `AddTable` ops are never produced, because
`internal_create_tables` creates all missing tables before the diff. This emitter
diverges from the runtime one: it emits FKs as **separate `ALTER TABLE ADD
FOREIGN KEY`** statements on Postgres (`emit.rs:175`) and **drops them with a
warning on SQLite** (`:249вЂ“256`) вЂ” a silent capability loss (AGENTS.md I-6 smell).

The migrate path already consumes the single producer correctly: Python
`compile_registry_schema_ir` (`src/ferro/ir/compiler.py:378`) в†’
`_set_schema_ir_modelset` over FFI (`src/ferro/__init__.py:121,154`) в†’
`SCHEMA_IR_MODELSET` (`src/state.rs:27`) в†’ `internal_migrate` diffs declared-IR vs
live. The CREATE path is the only declared-schema producer that does not consume
it.

The IR carries everything `build_column_plan` reads from JSON: `SchemaColumn` has
`logical_type` / `db_type` / `db_type_explicit` / `format` / `nullable` /
`primary_key` / `autoincrement` / `unique` / `index` / `default` / `enum_values`;
`SchemaModel` has `foreign_keys` / `indexes` / `uniques` / `checks` (pre-rendered).
**One exception** вЂ” the `bytes`/binary type вЂ” is handled in В§4.3.

## The change

### 4.1 One shared `SchemaModel в†’ CREATE TABLE` renderer (in `ferro-migrate`)

Promote `render_create_table` to a public entrypoint that emits the **complete**
create artifact set, with **inline foreign keys** folded in:

```rust
// crates/ferro-migrate/src/emit.rs  (public)
pub struct CreateTableEmission {
    pub create_sql: String,            // CREATE TABLE incl. inline FKs + inline single-col UNIQUE
    pub post_create_sqls: Vec<String>, // single + composite indexes, composite uniques, PG db_check
    pub warnings: Vec<String>,         // e.g. SQLite db_check elision
}
pub fn render_create_table(model: &SchemaModel, dialect: BackendDialect)
    -> Result<CreateTableEmission, EmissionError>;

pub fn order_models_for_create(models: &[SchemaModel]) -> Vec<&SchemaModel>;
```

Changes inside `ferro-migrate`:

- **Fold inline FKs into `render_create_table`** from `model.foreign_keys`,
  emitted anonymously (no `.name(...)`) to match the runtime bytes exactly, on
  both backends.
- **Rewire `emit_add_table_passes`** to call `render_create_table` and delete its
  separate `foreign_key_statements` pass (`emit.rs:175`) and the SQLite FK-drop
  warning (`:249вЂ“256`). This is the one intentional behavior change: the
  `AddTable` emitter now emits correct inline SQLite FKs.
- **Single topological sort.** `order_models_for_create` becomes the one
  FK-dependency ordering, replacing both `order_schemas_for_creation`
  (`schema.rs`, JSON) and the private `order_add_table_models` (`emit.rs`, IR).
- Keep the existing `single_column_unique_is_inline` (`emit.rs:113`) and
  `standalone_indexes` (`:127`) logic вЂ” they already reproduce the runtime's
  inline-single-unique / named-composite split correctly.

### 4.2 Runtime create path rewrite + wiring

`internal_create_tables` (`src/schema.rs:523`) stops reading `MODEL_REGISTRY` JSON
and reads the pushed `SCHEMA_IR_MODELSET`, exactly as `internal_migrate` does:

```rust
let modelset = SCHEMA_IR_MODELSET.read()? ...   // loud PyRuntimeError if absent (I-6)
for model in ferro_migrate::order_models_for_create(&modelset.payload.models) {
    let e = ferro_migrate::render_create_table(model, dialect)?;
    engine.execute_sql(&e.create_sql).await?;
    for sql in &e.post_create_sqls { engine.execute_sql(sql).await?; }
    for w in &e.warnings { crate::emit_user_warning(w); }
}
```

**Wiring requirement.** Every create entrypoint must push the modelset *before*
calling create. `connect`/`migrate` already do (`__init__.py:121,154`). The
standalone Python **`create_tables()`** path must be verified to push it (the
audit grep found `_set_schema_ir_modelset` only in the connect/migrate wrappers);
add the push if missing. A missing modelset fails loud вЂ” never a silent empty
create.

### 4.3 The `bytes`/binary fix (must land before deleting the legacy arm)

Removing `("string", Some("binary")) => Blob` would silently turn `bytes` columns
into `varchar`, because the compiler emits `logical_type="string"` for binary. So:

- **`src/ferro/ir/compiler.py::_logical_type`** (`:78`) вЂ” add a binary token:
  `if field_format == "binary": return "binary"`.
- **`canonical_from_parts`** вЂ” add `("binary", _) => Blob`.

The other four legacy arms (`date-time`, `date`, `uuid`, `decimal`) already have
domain-token equivalents the compiler emits (`datetime` / `date` / `uuid` /
`decimal`), so they are removed outright. Audit any other consumer of the schema
`logical_type` for the binary case (codec bind/fetch is a separate IR and should
be unaffected, but confirm).

```python
class Doc(Model):
    contents: bytes        # must remain BLOB (SQLite) / bytea (Postgres)
```

- **`src/schema.rs`** вЂ” `build_create_table_sqls` (`:453`), `canonical_column_type`

- **`src/migrate.rs`** вЂ” the `#[cfg(test)]` `schema_json_to_schema_ir` (`:1186`)

### 4.5 Re-point the parity test helper

`_render_create_table_sql_for_test` (`src/schema.rs:~618`) вЂ” the pyfunction the
Python cross-emitter parity tests call to obtain the Rust create-SQL вЂ” currently
takes a JSON schema and calls `build_create_table_sqls`. Re-point it through the
real IR path so the parity tests exercise what production does: the **Python**
side compiles the schema to a SchemaIR payload (`compile_schema_ir_payload`) and
passes that envelope; the **Rust** helper deserializes a `SchemaModel` and calls
`ferro_migrate::render_create_table`. `test_cross_emitter_parity.py` and
`test_db_type_cross_emitter_parity.py` then validate the IR-driven emitter
end-to-end.

## Parity details (the byte-for-byte risks)

The golden test (В§Testing) pins each of these:

1. **Column order.** `CREATE TABLE` column order must not move. The IR columns are
   alphabetical (`compile_schema_ir_payload` sorts `properties`); today's runtime
   iterates the serde JSON map. Confirm both orderings match (they should вЂ” both
   alphabetical) and pin it.
2. **Post-create artifact order.** Today the runtime emits per-column
   indexes/checks in column order, then composite uniques, then composite
   indexes. Exact statement-order parity is **infeasible from the IR**: the
   compiler sorts `indexes`/`uniques` by name and discards the user's
   declaration order, so the legacy order cannot be reconstructed for multiple
   same-kind composites. Because these are independent statements (every column
   exists before any post-create artifact runs), order is schema-irrelevant. The
   golden therefore asserts the **`CREATE TABLE` statement byte-exact** and the
   **post-create statements as a sorted set**; `test_cross_emitter_parity.py`
   (Alembic empty-diff) independently guarantees no schema-level difference.
3. **Anonymous inline FK.** The inline FK is emitted with no constraint name and
   `ON DELETE <action>` from `foreign_keys[].on_delete`, matching `schema.rs`
   today. The IR's `foreign_keys[].name` is **not** used for inline emission.
4. **Inline single-column UNIQUE.** `column.unique=true` в†’ inline `UNIQUE` on the
   column; the matching `uniques[]` single-column entry is skipped
   (`single_column_unique_is_inline`). Composite `uniques[]` (в‰Ґ2 cols) в†’ named
   `CREATE UNIQUE INDEX`.
5. **SQLite `db_check` elision.** `db_check` CHECK is emitted as a post-create
   `ALTER TABLE ADD CONSTRAINT` on Postgres only; on SQLite it is elided with a
   warning (unchanged from today).

## Error handling (decision: fail loud)

Today `canonical_column_type` ends in `.unwrap_or(CanonicalType::Varchar(None))` вЂ”
an **unknown column type silently becomes `varchar`**. #140 deliberately preserved
this (`Err в†’ Varchar(None)`) for behavior-parity. #153 **reverses it**: the
IR-driven path lets `render_create_table` propagate the `EmissionError` on an
unknown `logical_type` (and `internal_create_tables` surfaces it as a loud
`PyRuntimeError`). Rationale: silently creating a `varchar` for a type we do not
understand is the exact "degrade silently" anti-pattern AGENTS.md I-6 forbids, and
a well-formed model never produces an unknown type.

**Verification gate before adopting:** confirm no real model field compiles to the
compiler's `"unknown"` logical_type and relies on the varchar landing. If one
does, give that case a real `logical_type` in the compiler вЂ” do not retain a
catch-all silent fallback. (This is the only place #153 changes observable
behavior for malformed input, and it only ever turns a silent mis-type into a
clear error.)

All other failure modes are loud `PyResult` errors: missing modelset, unknown
table, emission failure. No `unwrap()` across the FFI boundary (I-3).

## Risk

The change touches the most-exercised DDL path. Principal risks and mitigations:

- **A subtle byte-difference in `CREATE TABLE`** (column order, FK rendering,
  artifact order). Mitigation: the golden byte-parity test captures current output
  *before* the refactor and asserts identity; В§Parity details enumerates every
  seam.
- **`bytes` regression** if the legacy arm is removed before the binary token
  lands. Mitigation: В§4.3 ordering + an explicit `bytes` parity case.
- **A create entrypoint not pushing the modelset.** Mitigation: loud failure (not
  silent empty create) + explicit verification of `create_tables()`.
- **FK consolidation changing `ferro-migrate` test expectations.** Expected and
  intentional (correctness improvement); the unit tests are updated in the same
  change.

## Migration impact

`none` вЂ” runtime DDL is byte-identical; the only behavior change is internal (the
test-only `AddTable` emitter gains correct SQLite FKs). The fail-loud reversal
(В§Error handling) only affects malformed models that cannot compile a valid type
today, turning a silent mis-type into a clear error.
