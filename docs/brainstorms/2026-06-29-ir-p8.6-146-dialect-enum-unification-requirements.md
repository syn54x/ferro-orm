# #146 — Unify the three dialect enums into one shared `Dialect`

## Goal

Collapse the **three parallel two-variant dialect enums** — one per crate layer,
kept in sync by **three hand-written per-seam translators** — into **one canonical
`Dialect`** defined in the leaf crate `ferro-ddl-lowering` and consumed everywhere
(runtime, lowering, migrate). Delete the three translators. This finishes the
"one type system, no parallel encoders" story for the dialect axis: cross-emitter
parity (AGENTS.md I-1) becomes **structural** rather than translator-mediated.

**This is a behavior-preserving, pure type-identity refactor. No emitted DDL
changes, no runtime behavior changes.** It is mechanical (compiler-driven rename
of the type name across ~270 reference sites) plus the relocation of one inherent
method (`from_url`) that cannot follow the type to a leaf crate.

This is sub-issue #146 of Epic #145 (Phase 8.6), surfaced during #140 (single
lowering library) and explicitly deferred past 8.5 / #153, which kept *adding*
translate-at-the-boundary helpers rather than unifying.

## The central design question (resolved: one type, not two)

`BackendKind` is not a pure DDL tag — it also carries **runtime identity**:
`from_url` (parses `sqlite:`/`postgres:` connection schemes), connection-pool
selection, query-execution dispatch, and introspection routing. So "which SQL
dialect do I render?" and "which database am I connected to?" are, in principle,
two responsibilities. The open question #146 poses: do they collapse into one
type, or does the runtime concern stay a distinct type bridged to the dialect?

**Decision: one type.** Rationale — the two concerns are the *same two-variant
axis*, 1:1 by construction, and that 1:1 is **bounded by the libraries** Ferro is
built on:

- DDL rendering is delegated to **`sea-query` 0.32**, whose dialect is a per-
  dialect builder: `SqliteQueryBuilder` / `PostgresQueryBuilder` /
  `MysqlQueryBuilder` — exactly three.
- Runtime drivers come from **`sqlx`**, which supports the same three.

Every value that can ever exist lines up one-to-one: SQLite↔SQLite,
Postgres↔Postgres, MySQL↔MySQL (MySQL is currently **rejected** at the door —
`from_url("mysql://…")` → `Err`, the only two `mysql` mentions in the codebase).
The classic driver-≠-dialect divergence cases do **not** arise in this stack:

- A second Postgres-wire engine (CockroachDB/Aurora) connects via the `postgres://`
  scheme and renders with `PostgresQueryBuilder` — it simply *is* "Postgres," not a
  new variant.
- An async/sync driver split (SQLAlchemy's `psycopg` vs `asyncpg`) cannot occur —
  `sqlx` is the single async driver per backend.

So a separate runtime type bridged by an **identity-mapped** translator carries
zero information and is exactly the hand-synced parallel abstraction this phase
removes. The runtime *behavior* (`from_url`, pools) is preserved not as a second
type but as behavior layered onto the one type in the crate that owns the runtime
(see §3.3). And the realistic future change — adding MySQL — gets *cheaper* under
one type: add one `Mysql` variant to one enum and the compiler's exhaustive-match
errors walk you to every site, versus adding it three times and re-threading three
translators.

## Non-goals (explicitly out of scope)

- **Splitting runtime-engine identity from DDL dialect into two types.** Rejected
  above — there is no divergence to model in the `sqlx` + `sea-query` stack. If a
  backend whose driver and dialect genuinely diverge is ever added, re-introducing
  the distinction is a localized, compiler-guided change at that time (YAGNI).
- **Adding MySQL (or any new backend).** Out of scope; mentioned only to show the
  unified type makes that future change cheaper.
- **Renaming the variants.** All three enums already use `Sqlite` / `Postgres`;
  only the *type name* unifies. No variant churn, no match-arm logic changes.
- **Changing any `sea-query` mapping logic.** The `match d { Sqlite => SqliteQueryBuilder,
  Postgres => PostgresQueryBuilder }` sites keep their logic; they just match on
  `Dialect` now.
- **The duplicated alter-helpers / legacy planner** (#142 / Phase 9) and any other
  8.6 sweep item.

## The change

### 3.1 Canonical type: promote `ferro_ddl_lowering::Dialect`

`ferro_ddl_lowering::Dialect` becomes the **one** dialect type. It gains
`#[derive(Default)]` with `#[default] Sqlite` to absorb `BackendKind`'s current
`Default`:

```rust
// crates/ferro-ddl-lowering/src/lib.rs
/// The one SQL dialect / database backend Ferro targets.
/// Selects both the rendered SQL dialect and (in the runtime crate) the driver.
#[derive(Clone, Copy, Debug, Default, PartialEq, Eq)]
pub enum Dialect {
    #[default]
    Sqlite,
    Postgres,
}
```

The unified derive set is the **union** of what the three sites need today:
`Clone, Copy, Debug, Default, PartialEq, Eq`. The plan must verify no remaining
site needs `Hash`/`Serialize`/`Deserialize` (none expected — dialect is passed as
a function argument, never serialized into the IR); add any that a real site
requires rather than working around it.

### 3.2 Delete the parallel enums and translators

- **main crate:** delete the `BackendKind` enum (`src/backend.rs:14`) and the
  `SqlDialect` alias (`src/state.rs:17`); delete `sql_dialect_to_lowering`
  (`src/schema.rs:168`) and `backend_dialect` (`src/migrate.rs:68`).
- **ferro-migrate:** delete `BackendDialect` (`crates/ferro-migrate/src/lib.rs:14`)
  and `lowering_dialect` (`crates/ferro-migrate/src/emit.rs:30`).
- **rename every reference** to `BackendKind` / `SqlDialect` / `BackendDialect`
  (~270 sites) to `Dialect`. Variants (`Sqlite`/`Postgres`) and all match-arm
  logic are unchanged; this is a pure type-name unification the compiler verifies.

### 3.3 Relocate `from_url` (the one non-mechanical piece)

`from_url` is the only inherent method on `BackendKind`, and inherent methods can
only be defined in the crate that defines the type — so it cannot become
`impl ferro_ddl_lowering::Dialect`. It moves to a **free function** in the main
crate (`src/backend.rs`), where parsing a connection-URL scheme already belongs:

```rust
// src/backend.rs — was `impl BackendKind { pub fn from_url(...) }`
pub fn dialect_from_url(url: &str) -> Result<Dialect, UnsupportedDatabaseUrl> {
    if url.starts_with("sqlite:") {
        Ok(Dialect::Sqlite)
    } else if url.starts_with("postgres://") || url.starts_with("postgresql://") {
        Ok(Dialect::Postgres)
    } else {
        Err(UnsupportedDatabaseUrl::from_url(url))
    }
}
```

`UnsupportedDatabaseUrl` and its exact message stay in the main crate, unchanged.
Callers flip from `BackendKind::from_url(url)` to `dialect_from_url(url)`: one
production caller (`src/connection.rs:199`) plus a handful of tests
(`src/backend.rs:765–788`). A free function (not a one-method extension trait) is
the right shape here — there is exactly one such constructor, so a trait would be
ceremony.

### 3.4 Re-exports and import paths (one definition, no re-declaration)

The canonical path is `ferro_ddl_lowering::Dialect`. To keep `ferro-migrate`'s
public API self-contained (its public fns — `render_create_table`, `plan_from_ir`,
`emit_sql_with_ir`, `order_models_for_create` — take a `Dialect`),
`ferro-migrate` **re-exports** it:

```rust
// crates/ferro-migrate/src/lib.rs
pub use ferro_ddl_lowering::Dialect;
```

The main crate imports `Dialect` from `ferro_ddl_lowering` (optionally via a single
`pub(crate) use` so its submodules share one path). The invariant: the enum is
**defined once**; everything else is `use`/`pub use`, never a second declaration or
alias.

## Error handling

No new failure modes. `dialect_from_url` preserves `from_url`'s exact
`Result<_, UnsupportedDatabaseUrl>` behavior and message. The translators being
deleted were infallible identity maps; removing them removes no error path. No
`unwrap()` is introduced across the FFI boundary (AGENTS.md I-3).

## Testing / validation

  `BackendDialect`, `sql_dialect_to_lowering`, `backend_dialect`, and
  `lowering_dialect` have **zero** occurrences anywhere in `src/` and `crates/`

  is gone; `dialect_from_url` is the sole URL→dialect entry.
- No **new** `dead_code` warnings introduced by the change.

**Done =** workspace `cargo test` green + the cross-emitter / auto-migrate pytest
matrix green on **SQLite and Postgres**, the three enums collapsed to one

and the grep-gate clean.

## Risk

- **Large mechanical diff (~270 sites).** Mitigation: it is a single type-name
  rename the compiler fully verifies — exhaustive `match` means any missed arm is a
  build error, not a silent bug. Land it as one focused change so the diff is
  reviewable as "rename + delete translators + relocate `from_url`," nothing else.
- **A behavior change sneaking in under the rename.** Mitigation: no match-arm
  logic or variant is touched; the full SQLite+Postgres matrix + cross-emitter
  parity assert byte-level invariance.
- **Derive mismatch** (e.g., a site relying on `Default`/`Hash` not on the
  canonical type). Mitigation: reconcile the derive set up front (§3.1); the
  compiler flags any missing derive immediately.
- **Merge friction with other in-flight 8.6 sub-issues.** Mitigation: the rename
  is mechanical, so rebasing later 8.6 work onto the unified name is a
  find-and-replace; keep this PR focused and land it promptly.

## Migration impact

`none` — internal architecture only. Runtime DDL and runtime behavior are
byte-/observably-identical; no user-facing API, type, or surface changes (the
unified type and `from_url` are internal Rust symbols, not part of the Python API).

`none` — no user-facing API, behavior, or surface change. Justification recorded
