# #155 — Stop swallowing the PEP 649 deferred-annotation eval error

## Goal

On **Python 3.14** (PEP 649 deferred annotations), when a model's annotations
fail to evaluate, `metaclass.py::_resolve_deferred_annotations` **swallows** the
real exception (`pass`) and returns an **empty** annotations dict. Pydantic then
treats the first class-body assignment as a bare non-annotated attribute and
raises a **misleading**:

```
PydanticUserError: A non-annotated attribute was detected: `id = 0`.
```

`id` is the symptom (the first assigned field); the lost exception is the cause.

**After this lands**, the both-formats-fail branch no longer swallows: it
attaches a model-naming note and **re-raises the real exception**
(`TypeError` / `NameError` / …), restoring the loud, accurate failure that
Python 3.13 already gives eagerly. No working model changes behaviour.

This is sub-issue #155 of Epic #145 (Phase 8.6). It is a **standalone metaclass
bug** (not a cross-crate duplication item), surfaced by the Python 3.14 / Pydantic
2.12 lane added in #141 (PR #152).

## The central design question (resolved): A — surface the error; **not** B — "make it work"

The issue offered two directions:

- **(A) Stop swallowing** — when both `__annotate_func__(1)` (VALUE) and `(2)`
  (FORWARDREF) fail and it is not the `BackRef[...]` special-case, re-raise the
  real error with context instead of `pass`.
- **(B) Make the failing models work** — diagnose why the deferred evaluation
  fails for `Annotated[T, FerroField(...)]` under PEP 649 and fix the eval path
  so the models that fail today actually work.

**Decision: (A) only. (B) is a non-goal because its premise is false** — see the
reframing finding below. The chosen shape of (A) is **(A2): re-raise with
context** (not a bare re-raise, not a special-cased "did you mean" hint).

### Reframing finding — (B) rests on a false premise (the decisive investigation result)

The issue's minimal repro is **invalid API usage**, not a valid model that PEP
649 breaks. There are two distinct `Field` symbols and the repro conflates them:

| Symbol | Origin | Declaration style | Has a `default` parameter? |
|--------|--------|-------------------|----------------------------|
| `ferro.Field(...)` → Pydantic `FieldInfo` | `ferro.fields` | assignment: `status: str = ferro.Field(default="draft")` | **yes** |
| `FerroField(...)` → `Annotated[...]` metadata | `ferro.base` | annotation: `Annotated[str, FerroField(...)]` | **no** — `(primary_key, autoincrement, unique, index, nullable, db_type, db_check)` |

So `Annotated[str, FerroField(default="draft")]` was **never valid** — `default`
belongs only to the assignment-side `ferro.Field`. A *valid* model with the same
shape works fine on 3.14:

```python
class C(Model):                                  # C.model_fields == ['id', 'status']  ✅ on 3.14
    id: Annotated[int, FerroField(primary_key=True)] = 0
    status: Annotated[str, FerroField(index=True)]
```

Verified live on `uv run python` (3.14.5):

- **The real swallowed error**, captured by calling `__annotate_func__` directly
  on the failing namespace:
  ```
  format 1 (VALUE):      TypeError: FerroField.__init__() got an unexpected keyword argument 'default'
  format 2 (FORWARDREF): TypeError: FerroField.__init__() got an unexpected keyword argument 'default'
  ```
  Both formats raise the **same** `TypeError` — the `Annotated[...]` metadata
  object is *constructed* in both VALUE and FORWARDREF evaluation, so an invalid
  constructor call fails identically in each. The `pass` then discards it.
- **`FerroField` inside `Annotated` evaluates correctly under PEP 649** (the `C`
  case above). The eval fails only when the annotation contains genuinely-broken
  code (an unsupported kwarg, an unquoted undefined name). There is **no
  valid-but-failing model** to "make work" — so (B) has nothing to fix.

This also means: on **3.13** the very same broken annotation raises the real
`TypeError` *eagerly at class-body execution*, before the metaclass runs. The
3.14 swallow is therefore a **developer-experience regression** — a clear error
became a misleading one — and (A2) restores 3.13 parity.

## Non-goals (explicitly out of scope — scoped-down, not hollowed-out, AGENTS.md I-6)

- **(B) "make the failing models work."** Removed: false premise (above). The
  models in the issue repro are invalid API usage; there is nothing valid to fix.
- **A targeted "did you mean `ferro.Field(default=...)`?" hint** (the rejected
  A3). Detecting the specific `FerroField(default=...)` mistake special-cases a
  *guess* about user intent — an AGENTS.md I-6 "best-effort" smell. The honest,
  general fix surfaces the real `TypeError`, which already names the offending
  call. No intent-guessing.
- **New forward-reference resolution machinery.** An *unquoted* forward reference
  (`x: list[NotDefinedYet]`) makes **both** formats raise `NameError` today and
  is silently swallowed → empty annotations. After this change it fails **loud**
  (`NameError` naming the model) — the intended fail-loud improvement, not a new
  feature. Quoted forward refs (`x: "NotDefinedYet"`) already work and are
  unaffected.
- **Broadening or narrowing the `BackRef[...]` special-cases.** Both
  `if "BackRef[...]" in str(...): raise` short-circuits (one per format) are
  preserved byte-for-byte. This issue touches only the terminal `pass`.
- **The eager (non-deferred) path.** The function's `return
  namespace.get("__annotations__", {})` fallthrough is reached only when there is
  no `__annotate_func__` (normal/eager classes). Unchanged.

## Current state (confirmed on `feat/ir-p8.6-cleanups`, Python 3.14.5)

`src/ferro/metaclass.py::_resolve_deferred_annotations`:

```python
if "__annotate_func__" in namespace and "__annotations__" not in namespace:
    try:
        return namespace["__annotate_func__"](1)        # VALUE (evaluate)
    except Exception as value_error:
        if "BackRef[...]" in str(value_error):
            raise value_error
        try:
            return namespace["__annotate_func__"](2)     # FORWARDREF
        except Exception as forward_error:
            if "BackRef[...]" in str(forward_error):
                raise forward_error
            pass                                         # <-- #155: swallows the real error

return namespace.get("__annotations__", {})              # <-- on 3.14: no __annotations__ -> {}
```

**Reproduced** (the issue's Case B) on 3.14.5:

```python
class B(Model):
    id: Annotated[int, FerroField(primary_key=True)] = 0
    status: Annotated[str, FerroField(default="draft")]   # invalid: FerroField has no `default`
# -> PydanticUserError: A non-annotated attribute was detected: `id = 0`.
```

**Proven safe** — instrumented the both-fail branch with a stderr marker and ran
the **entire suite + `examples/`**: **zero hits, 682 passed**. No currently-passing
model reaches the swallow, so nothing depends on the empty-dict fallback;
re-raising is behaviour-preserving for every model the suite exercises.

What pins the **old** swallow today (the one test that must flip):

- `tests/test_metaclass_internals.py::TestResolveDeferredAnnotations::test_annotate_func_both_fail`
  asserts `result == {}` when a mock `__annotate_func__` raises on both formats.
  This is the **unit-level seam** for the fix (see §Testing): it drives the
  both-fail branch with a mock, so it exercises the fixed code path on **3.13 and
  3.14 alike**, independent of the real PEP 649 machinery.

The sibling unit tests `test_annotate_func_format_1` and
`test_annotate_func_format_2_fallback` pin the success paths and must stay green
unchanged (they prove the two-format try is preserved).

## The change

### One source edit (`src/ferro/metaclass.py`, ~line 206)

Replace the terminal `pass` (and only that) with: attach a model-naming note to
the **VALUE-format** exception (`value_error` — "the original error", carrying the
traceback into the user's annotation line) and re-raise it.

```python
                except Exception as forward_error:
                    if "BackRef[...]" in str(forward_error):
                        raise forward_error
                    # #155: was `pass` -> returned {} -> Pydantic then raised the
                    # misleading "non-annotated attribute" on the first class-body
                    # assignment. Surface the REAL cause (PEP 649 deferred-eval
                    # failure) instead of silently dropping every annotation.
                    model = (
                        namespace.get("__qualname__")
                        or namespace.get("__name__")
                        or "<model>"
                    )
                    value_error.add_note(
                        f"ferro: could not resolve deferred annotations for model "
                        f"'{model}' (PEP 649 / Python 3.14). An annotation on this "
                        f"model failed to evaluate; an attribute Pydantic may flag "
                        f"as 'non-annotated' is a symptom, not the cause."
                    )
                    raise value_error
```

Rationale for the exact mechanics:

- **`add_note` (PEP 678, ≥3.11), not a wrapping exception.** Keeps the original
  exception's **type and traceback** intact (the most honest A2) and appends the
  diagnostic. No new exception type; no `raise ... from ...` double-traceback
  noise. (`forward_error` is in `__context__` automatically; that is acceptable
  and accurate — the FORWARDREF retry genuinely also failed.)
- **Re-raise `value_error`, not `forward_error`.** VALUE-format is the evaluation
  that matters for a model that should work; it is "the original error" the issue
  asks to surface. For the real triggers the two are the same type and message.
- **Nothing else in the function changes** — both `BackRef[...]`
  short-circuits, both `return` paths (format 1, format 2 success), and the eager
  fallthrough are untouched.

### One test comment correction (`tests/test_auto_migrate.py`, ~512–516)

The comment above `test_postgres_type_and_nullability_reconciliation`'s `MigPg`
model perpetuates the same misconception ("an `Annotated[T, FerroField(default=...)]`
sibling … makes deferred-annotation evaluation fail"). Correct it to state the
true cause — an *invalid / unevaluable* annotation is what fails, and the
assignment style sidesteps it by not deferring a broken expression — leaving the
model itself unchanged.

## Behaviour reconciliation (exactly what changes)

| Input on Python 3.14 | Today | After |
|----------------------|-------|-------|
| Valid model (`Annotated[T, FerroField(index=True)]` + assigned PK) | works | **works (unchanged)** |
| Unevaluable annotation (bad kwarg, unquoted forward ref) | `{}` → misleading `PydanticUserError: non-annotated 'id'` | **real `TypeError` / `NameError`** + model-naming note |
| `BackRef[...]` legacy syntax | raises (special-case) | raises (**unchanged**) |
| Eager / non-deferred class (no `__annotate_func__`) | returns `__annotations__` | returns `__annotations__` (**unchanged**) |

On **3.13** every row is already loud/eager and stays so; the only behavioural
delta lands on the 3.14 deferred path, and only for inputs that today silently
lose all annotations.

## Error handling

The new branch is purely additive failure surfacing: it converts a silent
empty-dict return into a re-raise of an exception that was already raised (and
caught) one line up — no new fallible operation, no `unwrap`-equivalent, no
broadened `except`. `add_note` cannot fail for a normal exception instance. The
two `BackRef[...]` guard-raises are preserved, so the legacy-syntax error path is
unchanged.

## Risk

- **A model that currently relies on the empty-dict fallback breaks.** Mitigation:
  instrumented full-suite + examples run shows **zero** swallow hits (682 passed)
  — no working model reaches the branch. The only test pinning the old behaviour
  is `test_annotate_func_both_fail`, updated intentionally.
- **Surfacing the wrong exception of the two.** Mitigation: `value_error`
  (VALUE-format, "the original") is chosen deliberately; `forward_error` remains
  visible via `__context__`. For the real triggers both are identical.
- **Wording churn in the note.** Low-stakes: the note is diagnostic prose, not a
  contract; tests assert only that it names the model, not its exact text.
- **3.13/3.14 asymmetry hides a gap.** Mitigation: the flipped unit test
  (`test_annotate_func_both_fail`) covers the fixed branch with a mock on **both**
  versions; the end-to-end test pins the user-facing contract on both; the 3.14
  CI lane covers the real deferred path.

## Migration impact

`none` — internal metaclass behaviour. Every working model is byte-for-byte
unaffected; the only change is that an annotation which **today** fails (silently,
with a misleading downstream error) **now** fails loudly with its real cause. No
public API, type, or surface change (`_resolve_deferred_annotations` is a private
static method).

## Documentation impact

`none` — no user-facing API, behaviour, or surface change for any valid model.
Justification recorded here per the roadmap documentation policy. (The corrected
`test_auto_migrate.py` comment is an internal accuracy fix, not user docs.)
